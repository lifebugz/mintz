# Release tooling â€” Design Document

**Date:** 2026-05-02
**Status:** Draft for review
**Scope:** versioning, CHANGELOG, GitHub Actions, npm publish for `mintz`
**Decision:** [Changesets](https://github.com/changesets/changesets) (vs. [release-please](https://github.com/googleapis/release-please))
**Distribution target:** npm (unscoped: `mintz` â€” confirmed available 2026-05-02 via `https://registry.npmjs.org/mintz` â†’ HTTP 404)

---

## 1. Overview

`mintz` is approaching its first public release. This document specifies how
versions, CHANGELOGs, GitHub releases, and npm publishes are produced â€”
end-to-end, with every step encoded in CI and one human gate (merging the
release PR).

The tool of record is **changesets**, configured for a single-package Bun
project. Stable releases run on every push to `main`; canary releases run on
demand via `workflow_dispatch`. A separate PR-time workflow enforces that
every PR contributes a changeset (or an explicit `--empty` marker for
no-release-needed PRs).

---

## 2. Decision: changesets, not release-please

Both tools solve the same problem (turn merged PRs into versioned npm
releases) and offer comparable automation: a bot opens and maintains a
"release PR"; merging it triggers publish; the only manual step is the
merge. They diverge on two axes.

### 2.1 Source of truth for release notes

| | release-please | changesets |
|---|---|---|
| Bump intent comes from | conventional-commit prefix | `.changeset/*.md` files (one per logical change) |
| CHANGELOG copy comes from | commit subjects | author-written prose blurbs |
| Per-PR overhead | none | ~30 seconds (`bunx changeset`) |
| Notes voice | scoped, terse, leaks implementation detail | curated user-facing, stable across refactors |

This project already practices strict conventional commits, which would make
release-please nearly free. **However**, commit grammar and release-notes
grammar serve different audiences:

- A commit message like `refactor(cli): replace fast-glob with native Bun.Glob`
  is correct internally but is *not* a release note (consumers don't care that
  fast-glob ever existed).
- A release note like "Reduces CLI cold-start by ~80ms via Bun.Glob native
  globbing" is the same change phrased for the consumer.

For a library that wants polished release notes from day one, the per-PR cost
of a written blurb is the right trade.

### 2.2 Pre-1.0 / snapshot ergonomics

`mintz` is shipping in 0.x for the foreseeable future. Changesets has
first-class support for snapshot releases (`changeset version --snapshot
canary` produces timestamp-based versions like `0.1.0-canary-20260502184517`
that publish under a non-default dist-tag). release-please's prerelease story
is awkward and configured per-major-version-line.

When (not if) we want to dogfood an unreleased fix in another project, we
need this. Building it in now is much cheaper than retrofitting later.

### 2.3 Out-of-scope blog post

A widely-cited blog post â€” [Setting up Changesets with Bun
Workspaces (Aug 2025)](https://ianm.com/posts/2025-08-18-setting-up-changesets-with-bun-workspaces) â€”
recommends substantial workarounds (`bun update` after `changeset version`,
per-package `bun publish || true` loop instead of `changeset publish`).

Those workarounds address a real, still-open issue:
[changesets/changesets#1468](https://github.com/changesets/changesets/issues/1468)
and [oven-sh/bun#24687](https://github.com/oven-sh/bun/issues/24687) â€”
changesets does not resolve `workspace:*` specifiers when publishing in Bun
monorepos, so dependent packages ship with literal `"workspace:*"` strings
that consumers cannot install.

**This problem does not apply to `mintz`:**

- `mintz` is a single package with no `workspace:*` references.
- The `examples/*` workspace entry in `package.json` is empty and exists only
  for future end-user examples (which won't depend on `mintz` via
  `workspace:*`; they'll depend on the published version after release).

We can therefore use the standard, vendor-documented changesets/action flow
unmodified, with one Bun-specific tweak: `bun publish` instead of
`npm publish`.

## 3. Architecture

### 3.1 Files added

```
.changeset/
  config.json            # changesets configuration
  README.md              # boilerplate, generated by `bunx changeset init`
  *.md                   # pending changesets (created per change worth releasing)

.github/
  workflows/
    release.yml          # stable release pipeline (push â†’ main)
    canary.yml           # manual canary publish (workflow_dispatch)
    changeset-check.yml  # PR-time enforcement that a changeset is included
```

### 3.2 Files modified

- `package.json` â€” three new scripts (`ci:version`, `ci:publish`, `ci:canary`)
  and one new devDependency (`@changesets/cli`).

No changes to `bunfig.toml`. No changes to `tsconfig.json` or
`bunup.config.ts`. No `prepublishOnly` lifecycle hook (build is invoked
explicitly inside `ci:publish` and `ci:canary` for visibility).

### 3.3 GitHub repository secrets required

- `NPM_TOKEN` â€” npm automation token with publish rights to the `mintz`
  package. Create from your npm account â†’ **Access Tokens** â†’ **Generate New
  Token** â†’ type **Automation** (bypasses 2FA from CI). Stored as a GitHub
  repository secret named `NPM_TOKEN`.
- `GITHUB_TOKEN` â€” provided automatically to every workflow run.

### 3.4 GitHub repository settings required

- Default branch: `main`. (Currently developing on `feat/v1`; a final merge
  back into `main` should precede setup of the release pipeline.)
- Branch protection on `main` (recommended, not required): require status
  check `Changeset Check / check` to pass on PRs; do not require reviews
  (solo dev). PR-time test gating is covered by an adjacent `ci.yml`
  workflow â€” see Section 5.4.

## 4. Configuration files

### 4.1 `.changeset/config.json`

Generated by `bunx changeset init` then trimmed to:

```json
{
  "$schema": "https://unpkg.com/@changesets/config@3.0.0/schema.json",
  "changelog": "@changesets/cli/changelog",
  "commit": false,
  "fixed": [],
  "linked": [],
  "access": "public",
  "baseBranch": "main",
  "updateInternalDependencies": "patch",
  "ignore": []
}
```

Rationale:

- `changelog: "@changesets/cli/changelog"` â€” built-in renderer; does not
  require GITHUB_TOKEN. The `@changesets/changelog-github` renderer adds
  PR-link rendering but is most useful for monorepos with many contributors.
  Switchable later without breaking history.
- `commit: false` â€” `changesets/action` handles git commits in the version
  PR; do not double-commit at `changeset version` time.
- `access: "public"` â€” required for unscoped public packages on npm. Also
  matches `publishConfig.access` in `package.json`.
- `baseBranch: "main"` â€” release branch. Used by `changeset status --since`
  to identify new changesets in PRs.
- `fixed: []`, `linked: []` â€” single-package; no internal sibling versions to
  pin together.
- `updateInternalDependencies: "patch"` â€” moot for single-package; left at
  default for future-proofing.

### 4.2 `package.json` additions

```json
{
  "scripts": {
    "changeset": "bunx changeset",
    "ci:version": "bunx changeset version",
    "ci:publish": "bun run build && bun publish --tolerate-republish && bunx changeset tag",
    "ci:canary": "bunx changeset version --snapshot canary && bun run build && bun publish --tag canary --tolerate-republish"
  },
  "devDependencies": {
    "@changesets/cli": "^2.27.0"
  }
}
```

Rationale:

- `bunx changeset` (not bare `changeset`): works whether or not a binary is
  symlinked into PATH; survives lockfile-only installs.
- `ci:publish` calls `bun publish` directly rather than `changeset publish`.
  `@changesets/cli` does not detect Bun via the lockfile (it understands
  npm/yarn/pnpm only) and would fall back to `npm publish`. For a Bun-native
  library, calling `bun publish` directly is consistent with the rest of
  the toolchain and inherits Bun's `--tolerate-republish` retry safety.
- `bunx changeset tag` after `bun publish` pushes the version git tag and
  creates the GitHub Release entry â€” those parts of `changeset publish` are
  package-manager-agnostic and worth keeping.
- Build runs *inside* `ci:publish` (not via a `prepublishOnly` lifecycle
  hook). Reason: explicit beats implicit. A local `bun publish --dry-run`
  does not surprise-trigger a rebuild of `dist/`.
- `bun publish --tolerate-republish` on both stable and canary paths:
  re-runs of a failed workflow will not error out if the version was
  actually pushed before the retry. Restartability matters more than strict
  idempotency detection here.
- `@changesets/cli` pinned in `bun.lock` rather than fetched on demand by
  `bunx`: reproducible CI; one less network call per release.

## 5. Workflow files

### 5.1 `.github/workflows/release.yml`

```yaml
name: Release

on:
  push:
    branches: [main]

# Only one release run at a time per branch â€” prevents two concurrent
# pushes from racing to open conflicting Version PRs.
concurrency: ${{ github.workflow }}-${{ github.ref }}

jobs:
  release:
    runs-on: ubuntu-latest
    permissions:
      contents: write          # tag + GitHub Release
      pull-requests: write     # open/update Version PR
      id-token: write          # reserved for future npm provenance support
    steps:
      - uses: actions/checkout@v6
        with: { fetch-depth: 0 }   # full history for changeset diffing

      - uses: oven-sh/setup-bun@v2

      - name: Configure npm auth
        run: echo "//registry.npmjs.org/:_authToken=${{ secrets.NPM_TOKEN }}" > "$HOME/.npmrc"

      - run: bun install --frozen-lockfile
      - run: bun run lint
      - run: bun run typecheck
      - run: bun test
      - run: bun run test:types

      - uses: changesets/action@v1
        with:
          version: bun run ci:version
          publish: bun run ci:publish
          commit: "chore: release"
          title: "chore: release"
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
          NPM_CONFIG_TOKEN: ${{ secrets.NPM_TOKEN }}
```

Notes:

- Pre-release checks (`lint`, `typecheck`, `test`, `test:types`) run *before*
  `changesets/action`. If they fail, neither the version PR nor the publish
  proceeds â€” broken code never ships.
- Both `.npmrc` (file-based) and `NPM_CONFIG_TOKEN` (env-based) are set:
  `bun publish` prefers the env var per the [Bun publish
  docs](https://bun.com/docs/pm/cli/publish), but some tooling still consults
  the file. Belt-and-suspenders is cheap.

### 5.2 `.github/workflows/canary.yml`

```yaml
name: Canary Release

on:
  workflow_dispatch:
    inputs:
      tag:
        description: "dist-tag for this canary"
        default: "canary"
        required: true

concurrency: ${{ github.workflow }}

jobs:
  canary:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      id-token: write
    steps:
      - uses: actions/checkout@v6
        with: { fetch-depth: 0 }

      - uses: oven-sh/setup-bun@v2

      - name: Configure npm auth
        run: echo "//registry.npmjs.org/:_authToken=${{ secrets.NPM_TOKEN }}" > "$HOME/.npmrc"

      - run: bun install --frozen-lockfile
      - run: bun test
      - run: bun run test:types

      - name: Publish snapshot
        run: |
          bunx changeset version --snapshot ${{ inputs.tag }}
          bun run build
          bun publish --tag ${{ inputs.tag }} --tolerate-republish
        env:
          NPM_CONFIG_TOKEN: ${{ secrets.NPM_TOKEN }}
```

Notes:

- Triggered by clicking "Run workflow" in the GitHub UI; the `tag` input
  defaults to `canary` but can be overridden (`next`, `experiment-foo`, etc.)
  per invocation.
- Does not commit version-bump changes back to git: the snapshot version is
  ephemeral by design, used only for the publish artifact.

### 5.3 `.github/workflows/changeset-check.yml`

```yaml
name: Changeset Check

on:
  pull_request:
    branches: [main]

jobs:
  check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v6
        with: { fetch-depth: 0 }

      - uses: oven-sh/setup-bun@v2

      - run: bun install --frozen-lockfile

      - name: Verify a changeset is present
        run: bunx changeset status --since=origin/${{ github.base_ref }}
```

Per the official [automating-changesets
docs](https://github.com/changesets/changesets/blob/main/docs/automating-changesets.md),
`changeset status --since=origin/main` exits non-zero if no new changesets
were added in the PR. For PRs that don't warrant a release entry (refactors,
docs, internal tests), the author runs `bunx changeset --empty` which creates
a no-bump marker file that satisfies the check.

No PR-title or label-based skip rules. The `--empty` flag *is* the official
escape hatch and trumps inferred-intent heuristics: explicit author signal
beats fragile pattern-matching on PR titles.

### 5.4 Adjacent: `ci.yml` (out of scope)

This spec covers release tooling only. A complementary `ci.yml` workflow
that runs `bun run lint`, `bun run typecheck`, `bun test`, and
`bun run test:types` on `pull_request` against `main` is recommended but
defined separately. Its check name (`CI / test`, by convention) should be
required by branch protection alongside `Changeset Check / check`.

`release.yml` re-runs the same checks before invoking `changesets/action`,
so a broken release path is impossible even if `ci.yml` is missing or
disabled â€” the duplication is intentional and provides defense in depth.

## 6. Developer flow

### 6.1 Per-PR ritual

For a change worth releasing (feature, bug fix, breaking change visible to
consumers):

```bash
bunx changeset
```

Interactive prompts:

1. **Bump level**: `patch` / `minor` / `major`.
2. **Summary**: free-form prose. Becomes the user-facing CHANGELOG entry.
   Multi-line accepted.

Example output (committed alongside the code in the same PR):

```markdown
<!-- .changeset/quiet-tigers-leap.md -->
---
"mintz": minor
---

Add `--watch` mode to the CLI for continuous regeneration on file changes.
Watches your tsconfig include paths and re-runs the transform with debounced
edits. Useful during active development to keep generated outputs in sync
with type changes.
```

For a change that does **not** warrant a release entry (refactor that doesn't
change behavior, docs, internal tests, CI tweaks):

```bash
bunx changeset --empty
```

Output: a no-bump marker file. Commit it; PR passes the check; no version
entry is produced.

### 6.2 Manual edits

The `.changeset/*.md` files are plain markdown with a YAML frontmatter
declaring bump levels. Authors who prefer to skip the prompt can write the
file directly. The frontmatter format:

```markdown
---
"<package-name>": <patch|minor|major>
---

<release-note prose>
```

For an empty marker: an empty frontmatter (`---\n---`) and no body.

## 7. CI/CD flow

### 7.1 Stable releases (push to `main`)

1. Maintainer merges feature PR into `main` (its `.changeset/*.md` files
   land along with the code).
2. `release.yml` triggers. Test/lint/type checks run.
3. `changesets/action` inspects pending changesets:
   - **If pending changesets exist:** the action opens or updates a PR
     titled `chore: release`. The PR commit:
     - Bumps `mintz` version in `package.json` to the highest declared bump
       level across all pending changesets (`major` > `minor` > `patch`).
     - Rewrites `CHANGELOG.md` consolidating the changesets' bodies into a
       single new section.
     - Deletes the consumed `.changeset/*.md` files.
     - Updates `bun.lock` (no-op for single-package; runs `bun install` as
       part of the action's standard flow).
   - **If no pending changesets exist** AND the previous push was the merge
     of a release PR: the action runs `ci:publish`, which:
     - Builds (`bun run build`).
     - Publishes via `bun publish --tolerate-republish` (Bun-native).
     - Creates the `vX.Y.Z` git tag locally via `bunx changeset tag`.
     - `changesets/action` pushes commits + tags as part of its post-publish
       git operations (it owns the git remote interaction; our `ci:publish`
       does not need its own `git push`).
     - `changesets/action` then attempts to create a GitHub Release entry
       from the published-package output. It expects npm-style
       `+ pkg@version` lines, which `bun publish` does emit, so this
       generally works. **Fallback:** if the GitHub Release isn't
       auto-created, the git tag still exists and a one-off
       `gh release create` step can be added (see Section 10.4).
   - **If no pending changesets exist and the previous push was unrelated
     (a chore commit, a doc edit, etc.):** no-op.
4. Repeat indefinitely. The release PR is recreated and re-updated as more
   feature PRs land between releases.

### 7.2 Canary releases (workflow_dispatch)

1. Maintainer clicks "Run workflow" on `canary.yml` in the GitHub UI.
2. Optionally overrides the `tag` input (default `canary`).
3. Pipeline runs:
   - Install + test + type-check.
   - `bunx changeset version --snapshot canary` produces version
     `<base>-canary-<timestamp>` (e.g., `0.1.0-canary-20260502184517`).
   - Build.
   - `bun publish --tag canary --tolerate-republish`.
4. `mintz@canary` is now installable via `bun add mintz@canary`. Stable
   `latest` dist-tag is unaffected.

### 7.3 PR checks (changeset-check.yml)

On every PR opened, synchronized, or reopened against `main`:

1. Checkout, setup-bun, install.
2. `bunx changeset status --since=origin/main` â€” exits non-zero if no
   changesets were added in this PR.
3. Author resolves a failure by either running `bunx changeset` (for
   release-relevant PRs) or `bunx changeset --empty` (for no-release PRs)
   and pushing the resulting `.changeset/*.md` file.

## 8. Initial release: cutting v0.1.0

### 8.1 Strategy

Write **one curated changeset** summarizing the initial release rather than
attempting to reconstruct per-commit changesets retroactively from the 43
existing commits. The pre-release commits include intermediate refactors and
test-only PRs that shouldn't appear in the v0.1.0 release notes.

### 8.2 Concrete steps

After all of Section 4 and Section 5 land in `main`:

```bash
# 1. Initialize changesets if not yet done
bunx changeset init

# 2. Trim/replace the generated config to match Section 4.1

# 3. Write the v0.1.0 changeset
bunx changeset
# Pick: minor (will produce 0.0.0 â†’ 0.1.0)
# Summary: see template below
```

Suggested summary text:

```markdown
---
"mintz": minor
---

Initial public release of `mintz`.

`mintz` reads your TypeScript types at runtime â€” Bun-native, with zero magic
at the call site.

### Engine
- Resolves generic type parameters to concrete literal values via the
  TypeScript checker
- Deterministic literal sort and dedup for byte-stable output
- Find / resolve / classify / emit pipeline integrated into `transform()`

### Bun integration
- Plugin factory for Bun build mode and `bunfig.toml` `preload` mode
- Preload mode rewrites `mint<T>()` call sites at module-load time

### CLI
- `mintz check` â€” drift detection (exits non-zero if generated output
  differs from what the transform would emit; suitable for CI gating)
- `mintz` (default) â€” rewrite mode with atomic writes
- `mintz --watch` â€” continuous regeneration on file changes
```

Then commit, open a PR, merge, and let `release.yml` open the v0.1.0 release
PR. Merge that. v0.1.0 lands on npm.

### 8.3 Why curated rather than auto-generated

A naive auto-generation from existing commits would produce a CHANGELOG
that reads like a development log: "test: fixture harness with string-union
case", "docs(plan): align error fixture codes with actual engine behavior",
etc. These are correct internal records but are not what an adopter looking
at v0.1.0 wants to read.

Future releases (v0.2.0+) will not have this problem: per-PR changesets will
accumulate organically, and the CHANGELOG will be precise without curation.
The v0.1.0 step is a one-time event.

## 9. Edge cases & error handling

### 9.1 Publish fails partway

The release PR has been merged â€” `package.json` shows the new version, but
nothing reached npm (network blip, NPM_TOKEN expired, etc.).

**Recovery:** re-run the failed workflow from the GitHub Actions UI. Both
the stable (`ci:publish`) and canary (`ci:canary`) paths use
`bun publish --tolerate-republish`, so a re-run that finds the package
already published exits 0 instead of erroring. `bunx changeset tag` is also
idempotent â€” re-creating an existing git tag is a no-op locally and a
fast-forward push is harmless if the tag already exists upstream.

### 9.2 Two PRs merge in quick succession

The `concurrency:` directive in `release.yml` ensures only one release run
is active per branch. The second push enqueues; when it runs, the version
PR is updated to bundle both PRs' changesets into a single release.

### 9.3 A changeset declares `major` while in 0.x

In 0.x, changesets's default behavior treats `major` bumps as `0.x â†’
0.(x+1)` (semver minor). To force a true `1.0.0`, manually bump
`package.json` and tag â€” out of scope for this design. **Decision:** during
0.x development, `major` changesets are equivalent to `minor` bumps to the
next 0.x line. The promotion to `1.0.0` is a deliberate manual milestone.

### 9.4 Canary publish before any stable release

`bunx changeset version --snapshot canary` operates on whatever version
`package.json` currently declares. From `0.0.0`, it produces
`0.0.0-canary-<timestamp>`. Useful for end-to-end testing of the publish
pipeline before v0.1.0.

### 9.5 Lockfile drift

If `package.json` is edited but `bun.lock` is not regenerated,
`bun install --frozen-lockfile` fails. This blocks every CI workflow,
including release. The `changeset-check.yml` job catches this on PR (it
also runs `bun install --frozen-lockfile`); the contributor must regenerate
the lockfile and push.

### 9.6 npm name conflict

Pre-flight check completed 2026-05-02:
`https://registry.npmjs.org/mintz` returns HTTP 404 and `bun info mintz`
confirms the package does not exist. Name is available for unscoped
publish. (Note: npm reserves names for 24 hours after unpublish; a 404 is
*usually* "available" but technically "not currently in catalog". For
`mintz`, no prior history exists.)

### 9.7 Force-push to `main`

Rare given solo dev + recommended branch protection. `changesets/action`
recovers gracefully â€” it operates on the current `main` and reopens the
version PR if needed. No mutable state in the workflow itself.

## 10. Open questions and future work

### 10.1 npm provenance

[npm provenance](https://docs.npmjs.com/generating-provenance-statements)
provides cryptographic attestation that a package was built from a specific
git commit in a specific GitHub Actions run. The `id-token: write` permission
is already declared in `release.yml` and `canary.yml` to leave the door open.

`bun publish` does not currently support provenance attestations (as of
2026-05). When it does, add `--provenance` to the publish step. No design
change required now.

### 10.2 JSR mirror publish

The original project design (2026-04-30 spec) mentions JSR as a v2
distribution target. JSR has its own publish flow (`bunx jsr publish` or
`deno publish`); when adopted, add a step to `release.yml` after
`changesets/action` that runs the JSR publish on the same release commit.
Out of scope for this document.

### 10.3 Optional changeset-bot

The official changesets automation guidance recommends the
[changeset-bot](https://github.com/apps/changeset-bot) GitHub App as a
non-blocking complement to the blocking CI check. The bot comments on PRs
without changesets and provides a one-click "Add changeset" UI.

For solo development, the bot is largely redundant. If `mintz` ever accepts
external contributions, install the bot â€” contributors can add changesets
without local Bun installed. One-line cost: install the App on the repo.

### 10.4 GitHub Release auto-creation fallback

`changesets/action`'s GitHub Release auto-creation depends on parsing
publish-command output for `+ pkg@version` lines. `bun publish` emits this
format (per the [Bun publish docs](https://bun.com/docs/pm/cli/publish)), so
auto-creation is expected to work. If a release ships with a git tag but no
GitHub Release entry, add this step to `release.yml` after the
`changesets/action` step:

```yaml
- name: Create GitHub Release if missing
  if: steps.changesets.outputs.published == 'true'
  env:
    GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}
  run: |
    VERSION=$(jq -r .version package.json)
    gh release view "v$VERSION" >/dev/null 2>&1 || \
      gh release create "v$VERSION" --notes-file CHANGELOG.md --title "v$VERSION"
```

This is idempotent: it skips when the Release already exists.

### 10.5 Migration to release-please

If the curation overhead of changesets ever proves too high, migrating to
release-please is straightforward (both tools are PR-driven; the migration
would replace `.changeset/` and the workflow files with
`release-please-config.json` and a different action). No version-history
loss either way.
