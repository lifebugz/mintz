# Release Tooling Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Set up the changesets-based release pipeline for `mintz`, enabling
automated stable releases on push to `main`, manual canary releases via
`workflow_dispatch`, and PR-time enforcement that every change includes a
changeset (or an explicit `--empty` marker). Cut v0.1.0 as the first release.

**Architecture:** Single-package changesets configuration; four GitHub Actions
workflows (`release.yml` for stable releases, `canary.yml` for manual
snapshots, `changeset-check.yml` for PR enforcement, `ci.yml` for PR-time
lint/test/typecheck); `package.json` scripts wire build + `bun publish` +
changeset-tag together. Bun-native publish (no `changeset publish` fallback
to npm).

**Tech Stack:** `@changesets/cli` ≥ 2.27, `changesets/action@v1`,
`oven-sh/setup-bun@v2`, `actions/checkout@v6`, `bun publish`, GitHub Actions.

**Scope discipline:** This plan does not modify library source (`src/`), test
code (`test/`), build config (`bunup.config.ts`, `tsconfig.json`), lint
(`eslint.config.mjs`), or formatter (`.prettierrc.json`). The only change to
the existing tree is adding scripts + one devDependency to `package.json`
and adding new files under `.changeset/` and `.github/workflows/`.

**Spec reference:** `docs/superpowers/specs/2026-05-02-release-tooling-design.md`
(committed as `66ef486` on branch `feat/v1`).

## Phase 2 — Workflow files

### Task 4: Create `.github/workflows/release.yml`

**Files:**
- Create: `.github/workflows/release.yml`

- [ ] **Step 1: Confirm `.github/workflows/` does not yet exist**

```bash
ls -la .github/workflows/ 2>/dev/null || echo "directory does not exist"
```

Expected: prints "directory does not exist". (Confirms we're starting
clean.)

- [ ] **Step 2: Create the file with this content**

Path: `.github/workflows/release.yml`

```yaml
name: Release

on:
  push:
    branches: [main]

# Only one release run at a time per branch — prevents two concurrent
# pushes from racing to open conflicting Version PRs.
concurrency: ${{ github.workflow }}-${{ github.ref }}

jobs:
  release:
    runs-on: ubuntu-latest
    permissions:
      contents: write          # tag + GitHub Release
      pull-requests: write     # open/update Version PR
      id-token: write          # reserved for future npm provenance support
    steps:
      - uses: actions/checkout@v6
        with:
          fetch-depth: 0

      - uses: oven-sh/setup-bun@v2

      - name: Configure npm auth
        run: echo "//registry.npmjs.org/:_authToken=${{ secrets.NPM_TOKEN }}" > "$HOME/.npmrc"

      - run: bun install --frozen-lockfile
      - run: bun run lint
      - run: bun run typecheck
      - run: bun test
      - run: bun run test:types

      - uses: changesets/action@v1
        with:
          version: bun run ci:version
          publish: bun run ci:publish
          commit: "chore: release"
          title: "chore: release"
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
          NPM_CONFIG_TOKEN: ${{ secrets.NPM_TOKEN }}
```

- [ ] **Step 3: Validate YAML syntax**

```bash

Expected: prints `valid YAML`. (If `js-yaml` is unavailable, fall back to

on:
  workflow_dispatch:
    inputs:
      tag:
        description: "dist-tag for this canary"
        default: "canary"
        required: true

concurrency: ${{ github.workflow }}

jobs:
  canary:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      id-token: write
    steps:
      - uses: actions/checkout@v6
        with:
          fetch-depth: 0

      - uses: oven-sh/setup-bun@v2

      - name: Configure npm auth
        run: echo "//registry.npmjs.org/:_authToken=${{ secrets.NPM_TOKEN }}" > "$HOME/.npmrc"

      - run: bun install --frozen-lockfile
      - run: bun test
      - run: bun run test:types

      - name: Publish snapshot
        run: |
          bunx changeset version --snapshot ${{ inputs.tag }}
          bun run build
          bun publish --tag ${{ inputs.tag }} --tolerate-republish
        env:
          NPM_CONFIG_TOKEN: ${{ secrets.NPM_TOKEN }}
```

## Phase 3 — v0.1.0 changeset

### Task 8: Write the curated v0.1.0 changeset

**Files:**
- Create: `.changeset/initial-release.md`

The filename is hand-chosen (rather than the random `quiet-tigers-leap.md`
style produced by `bunx changeset`) so its purpose is self-documenting in
git history.

- [ ] **Step 1: Create the file with this content**

Path: `.changeset/initial-release.md`

```markdown
---
"mintz": minor
---

Initial public release of `mintz`.

`mintz` reads your TypeScript types at runtime — Bun-native, with zero magic
at the call site.

### Engine
- Resolves generic type parameters to concrete literal values via the
  TypeScript checker
- Deterministic literal sort and dedup for byte-stable output
- Find / resolve / classify / emit pipeline integrated into `transform()`

### Bun integration
- Plugin factory for Bun build mode and `bunfig.toml` `preload` mode
- Preload mode rewrites `mint<T>()` call sites at module-load time

### CLI
- `mintz check` — drift detection (exits non-zero if generated output
  differs from what the transform would emit; suitable for CI gating)
- `mintz` (default) — rewrite mode with atomic writes
- `mintz --watch` — continuous regeneration on file changes
```

- [ ] **Step 2: Verify the changeset is detected and produces the right bump**

```bash
bunx changeset status --verbose
```

Expected: prints something like:

```
🦋  Releases
🦋  mintz 0.0.0 -> 0.1.0 (minor)
```

Confirms the changeset was parsed correctly and will produce the right
version bump. (Do NOT run `bunx changeset version` here — that would consume
the changeset file and write `package.json` + `CHANGELOG.md` changes; we
want the version PR in CI to perform that step on `main`, not locally on
`feat/v1`.)

   this via repo → **Settings** → **Secrets and variables** → **Actions**).
2. Click **New repository secret**.
3. **Name:** `NPM_TOKEN` (exact case; the workflows reference this name).
4. **Value:** paste the token from Step 1.

## Appendix A — Rollback procedures

If something goes wrong mid-release:

**Wrong version published:** `npm` does not allow republishing the same
version. Bump again and publish a new version:

```bash
# add a patch changeset
bunx changeset
# pick patch; summarize the fix
# commit, open PR, merge, let release.yml fire
```

**Wrong content in a published version:** within 72 hours of publish, you
can `bun publish` over with `--tolerate-republish`... no, you can't.
Republish requires a new version. Use `npm deprecate mintz@<bad-version>
"<reason>"` to warn installers and publish a corrected new version.

**Need to fully unpublish (within 72 hours only):**

```bash
npm unpublish mintz@<version>
```

After 72 hours, only `npm deprecate` is available. Reserve unpublish for
true emergencies (leaked secrets, broken installs); deprecate is the
preferred tool.

**CI workflow itself is broken (e.g., `release.yml` won't run):** add the
`no-changeset` label to the fix PR — but this only skips `changeset-check`,
not the full release pipeline. To bypass the release pipeline entirely
during emergency hotfixes, manually disable `release.yml` from the GitHub
Actions UI temporarily; re-enable after the fix is in.
